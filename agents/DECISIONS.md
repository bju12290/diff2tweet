# important decisions only

- 2026-03-08: The initial config scaffold lives at the repo root as `diff2tweet.yaml` so the future CLI can use a simple default discovery path.
- 2026-03-08: API keys (e.g. `OPENAI_API_KEY`) must be supplied via environment variables or a `.env` file. They must never appear in `diff2tweet.yaml` or be committed to source control.
- 2026-03-08: CLI is invoked as `diff2tweet` with no arguments. The tool auto-discovers the git diff from the current repo and optionally reads `NOTES.md` from the repo root. Users never pass a diff file path manually.
- 2026-03-08: The output folder (`.diff2tweet/`) is gitignored by default. Users may opt in to committing logs by removing the `.diff2tweet/` entry from their `.gitignore`.
- 2026-03-08: Config implementation uses Pydantic v2 `BaseModel` for YAML, `pydantic-settings` `BaseSettings` for secrets, and a merged runtime config object for CLI consumption.
- 2026-03-08: Provider-specific validation happens at load time: the selected provider must have its corresponding API key in env / `.env`.
- 2026-03-08: Git diff source is committed changes only (`git diff <range>`). Staged changes, working tree changes, and untracked files are excluded - the tool is for tweeting about finished work.
- 2026-03-08: Default commit range is "since last run" (last-processed SHA stored in run log). First-run fallback is `lookback_commits` from config (default 5, i.e. `HEAD~5..HEAD`). Commit messages always span the same range as the diff.
- 2026-03-08: Run history for git discovery reads `.diff2tweet/run_log.jsonl` and uses the most recent JSON line with `last_processed_sha` as the prior processed commit.
- 2026-03-08: If the detected commit range is empty (no commits found), the tool raises a clear user-facing error rather than silently producing an empty result.
- 2026-03-08: Notes discovery walks up from the current working directory to the git repo root and treats a missing `NOTES.md` file as a non-error.
- 2026-03-08: Typer is the CLI framework. Chosen for ergonomic decorator API and clean support for future flags (`--lookback`, `--config`, `--dry-run`, etc.).
- 2026-03-08: `README.md` is included as LLM context by discovering it at the repo root and truncating to `readme_max_chars` (default 2000). No separate summary file - truncation naturally captures the front-loaded important content. Users set `readme_max_chars: 0` to exclude it.
- 2026-03-08: Git commit messages use `--format=%B` (full message body) to give the LLM maximum signal, including context developers write in commit message bodies.
- 2026-03-08: Config discovery is fixed at `<repo-root>/diff2tweet.yaml`, and discovery/config/provider/logging failures are converted into clean exit-code-1 CLI errors without tracebacks.
- 2026-03-08: Provider abstraction uses an abstract base class `BaseProvider` in `providers/base.py` with a single `generate_tweets(prompt_text: str, config: RuntimeConfig) -> list[str]` method. Concrete implementations live in `providers/openai_provider.py` (and later `providers/anthropic.py`, etc.). A `get_provider(config: RuntimeConfig) -> BaseProvider` factory in `providers/__init__.py` dispatches by `config.provider`.
- 2026-03-08: Generate exactly `num_candidates` tweet candidates per run (default 3, minimum 1, maximum 10).
- 2026-03-08: LLM prompt context is assembled in priority order: (1) README up to `readme_max_chars`, (2) all commit messages, (3) all NOTES.md content, (4) filtered diff filling the remainder up to `context_max_chars`.
- 2026-03-08: Before applying the context budget, diff text is filtered to strip noise (lock files, minified assets, generated files) using a configurable `diff_ignore_patterns` list. Filtering happens in `prompt.py`, not `git.py` - git discovery returns the full diff. Default patterns: `["*.lock", "package-lock.json", "yarn.lock", "poetry.lock", "Pipfile.lock", "*.min.js", "*.min.css", "dist/**", "build/**"]`.
- 2026-03-08: The OpenAI provider uses the chat-completions API and requires the model to return JSON with a top-level `tweets` array so the CLI can validate and log exactly `config.num_candidates` candidates.
- 2026-03-08: Successful generation runs append a JSONL record containing UTC timestamp, current HEAD SHA as `last_processed_sha`, commit range, and generated tweet candidates to `<output_folder>/run_log.jsonl`.
- 2026-03-08: `last_processed_sha` is written at generation time, not at approval time. Running the tool means "I have reviewed these commits." Approval status is recorded as additional data but does not gate SHA advancement. Denied-everything runs still advance the range so the same diff is never reviewed twice.
- 2026-03-08: Approval is collected synchronously in the CLI after all generated candidates are printed. Each candidate gets a y/n prompt in sequence. Approval statuses and a single approval timestamp are written by appending a second JSONL entry to the run log referencing the generation timestamp, keeping the log append-only.
- 2026-03-08: Tweet candidate count is configurable via `num_candidates: int` in `diff2tweet.yaml` (default 3, ge=1, le=10). Enforcement lives at the provider level: `BaseProvider.generate_tweets` docstring specifies that implementations must return exactly `config.num_candidates` candidates, and each concrete provider validates the response count against `config.num_candidates`. The `generate_tweets(prompt_text, config)` signature already carries `config`, so no signature change is needed.
- 2026-03-08: Markdown review artifacts live at `<output_folder>/runs/` and use a filename-safe version of the generation timestamp on Windows while preserving the original ISO timestamp inside the file contents and JSONL log.
- 2026-03-08: Raw README content is replaced by four structured project context fields in `diff2tweet.yaml`: `project_name` (str), `project_summary` (str), `project_audience` (str), `project_key_terms` (list[str]). All four are optional with empty defaults so existing configs degrade gracefully. They render as a single `## Project` section in the prompt instead of `## README`, giving the LLM clean, actionable framing with no noise.
- 2026-03-08: `readme_max_chars` default changes from 2000 to 0 (disabled). The structured project fields serve the same purpose better. The field is kept for power users who want both, but README context is opt-in from this point forward.
- 2026-03-08: `project_key_terms` is distinct from `forced_hashtags`. Key terms are topic vocabulary that informs the LLM's understanding of the project domain (e.g. "CLI", "open source", "git workflow"). Forced hashtags are literal strings that should appear in the tweet output (e.g. "#buildinpublic"). Both feed the prompt but serve different purposes.
- 2026-03-08: Context priority order updated to: project fields -> commit messages -> NOTES.md -> filtered diff (remainder of `context_max_chars`). README slot removed; project fields are always included in full before the budget is applied (they are small by design).
